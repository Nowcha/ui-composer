# ADR-003: ツリーdiffアルゴリズムと差分プロンプト

- 日付: 2026-07-12(Phase 2)
- ステータス: 採用

## コンテキスト

差分プロンプト(Phase 2の本丸)の基盤となるツリーdiff。docs/03 で
Opus設計ポイントに指定されていた箇所。

## 決定事項

1. **ID索引の完全一致方式**: ノードIDは編集をまたいで安定(再採番しない)
   という不変条件を前提に、両ツリーを `Map<id, {node, parentId, index}>` に
   展開して突き合わせる。ヒューリスティックなツリーマッチング(編集距離等)は不要
2. **追加/削除は最上位ノードのみ報告**: サブツリーごと追加された場合、
   子孫は暗黙に含まれるため個別報告しない(プロンプトのノイズ削減)
3. **移動の誤検知防止**: 兄弟の追加/削除によるインデックスずれは移動ではない。
   同一親内の移動は「生存ノード(両版に存在)同士の相対順序」が変わった場合のみ
   報告する。親が変わった場合は無条件で移動
4. **変更はフィールド粒度**(`props` / `icon` / `behavior` / `frozen` / `type`)で
   検出し、propsはキー単位の before → after を差分プロンプトに出力
5. 差分プロンプトには**凍結要素の再掲**を含める(差分に含まれても凍結優先を明示)
6. 比較は `JSON.stringify` ベースの値比較。SpecTreeはJSON安全なデータのみを
   持つ設計(storeにJSXを入れない規約)なのでこれで十分

## 却下した代替案

- **jsondiffpatch等の汎用diffライブラリ**: 出力が汎用パッチ形式で、
  「AIへの日本語指示」への変換コストが高い。ドメイン特化の構造化diffの方が
  プロンプト品質を制御できる
- **LIS(最長増加部分列)による最小移動集合の検出**: 移動報告が数個多くても
  プロンプトの実害がないため、実装コストに見合わない
